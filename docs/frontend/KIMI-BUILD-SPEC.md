## 1. Product Summary

Box Seat is a live/replay **match-story visualizer** for the 2026 World Cup, built
for the TxODDS x Solana hackathon (Consumer and Fan Experiences track). It renders a
single match as a 3D scene: an accurate football pitch with a glowing, team-colored
**pressure terrain** rising above it (who is applying pressure, and where on the pitch),
and a **momentum ribbon** flowing above the terrain that encodes momentum-over-time and
doubles as the replay scrub control. Goals, red cards, and VAR overturns interrupt the
ambient scene with a **full-screen broadcast-style takeover**. A minimal 2D **scorebug
HUD** (badges, score, clock, event log) overlays the scene. The same engine serves two
temporal modes: **live** (state streams in over WebSocket) and **replay** (finished
matches are fully scrubbable). It is what a good commentator narrates ‚Äî "they're really
building pressure now" ‚Äî made visible and precise.

The frontend is a **pure viewer**. All data comes from one WebSocket exposed by our own
backend. The backend handles all TxLINE ingestion and all Solana activity.

## 2. Hard Constraints (non-negotiable)

1. **No betting, odds, or predictions ‚Äî anywhere, ever.** No odds display, no "win
   probability", no implied-probability framing, no "who will win" UI, no prediction
   games. This is an ethical boundary set by the founder, not a style preference. The
   backend never sends odds data, so this should be structurally impossible ‚Äî but do
   not invent any feature in this space, even as flavor text. Momentum is a description
   of the run of play so far, never a forecast. Never label anything "chance of
   winning", "likely", "predicted", etc.
2. **No wallet UI, no crypto UI, zero Solana surface in the frontend.** No wallet
   connect button, no addresses, no transaction links, no token anything. Solana lives
   entirely in the backend behind a service wallet. The frontend never imports a Solana
   library.
3. **No direct TxLINE calls from the frontend.** The one and only data source is the
   backend WebSocket described in section 4. No other network calls for match data.
4. **Mostly 3D.** 2D is reserved strictly for numbers/text as a HUD overlay (score,
   team names/badges, clock, event log, transport controls) and for the sanctioned
   full-screen key-moment takeover. Everything else ‚Äî including the momentum-over-time
   visualization ‚Äî is 3D, inside the one scene. Do not build a 2D momentum chart, 2D
   pressure bars, or any dashboard-style panels.

## 3. Screens and Interaction Structure

Two routes plus one overlay state. Nothing else.

```
/                      Match list (functional entry point, not a centerpiece)
/match/:fixtureId      Match view ‚Äî the 3D scene. Serves both live and replay;
                       switches to replay automatically when the match is finalised.
   ‚îî‚îÄ Key-moment takeover ‚Äî full-screen overlay state inside match view (not a route)
```

Interaction model is deliberately minimal:

- **Everywhere:** orbit/zoom the camera (drag + wheel/pinch), double-click to reset
  camera.
- **Replay only:** scrub by pointing/dragging on the momentum ribbon; small HUD
  transport controls (play/pause, speed, current minute); optional cinematic fly-along
  toggle.
- **Takeover:** any click or Escape dismisses it early.

No other controls. No settings, no filters, no share buttons.

## 4. Backend Contract (build against this exactly)

One WebSocket endpoint. Default `ws://localhost:8787` in dev; read the URL from an env
var (e.g. `VITE_WS_URL` / `NEXT_PUBLIC_WS_URL`) so production can point at `wss://...`.

### Client ‚Üí Server

```json
{ "type": "subscribe", "fixtureId": 14790158 }
{ "type": "unsubscribe", "fixtureId": 14790158 }
```

A client may hold multiple subscriptions simultaneously (used by the match list).

### Server ‚Üí Client

**`state`** ‚Äî sent on every backend reducer update for a subscribed fixture. Each
message is the **full authoritative current state**, not a delta. Replace, don't merge.

```json
{
  "type": "state",
  "state": {
    "fixtureId": 14790158,
    "statusId": 4,
    "score": { "participant1": 1, "participant2": 0 },
    "momentum": 0.42,
    "pressure": {
      "participant1": { "defensive": 3, "middle": 8, "attacking": 14 },
      "participant2": { "defensive": 5, "middle": 4, "attacking": 2 }
    },
    "keyMoments": [
      { "type": "goal", "participant": 1, "ts": 1721300123456, "seq": 42 }
    ],
    "lastTs": 1721300456789,
    "lastSeq": 118
  }
}
```

Field semantics:

- `momentum`: number in `[-1, +1]`. **Positive favors participant1, negative favors
  participant2.** 0 is neutral.
- `pressure.participantN.{defensive,middle,attacking}`: unbounded accumulating totals
  per pitch third, from that team's own perspective (`attacking` = the third nearest
  the opponent's goal). **Normalize on the frontend** (section 8.4).
- `keyMoments`: the full list to date, in order. `type` is one of `"goal"`,
  `"red_card"`, `"var_overturned"`. `participant` is `1` or `2`. Detect *new* moments
  by tracking the highest `seq` you have already handled (not just array length ‚Äî
  robust against reconnects re-sending state).
- `statusId`: `1` = scheduled, `100` = finalised (`game_finalised`). Intermediate
  in-play codes are not guaranteed to be a fixed set ‚Äî treat *anything other than 1 or
  100* as "in progress". Never switch on unknown status codes.
- `ts`/`lastTs` are epoch milliseconds; `seq` is a monotonically increasing
  per-fixture sequence number.

**`replay_chunk`** ‚Äî sent after you subscribe to a **finalised** fixture. The backend
streams the fixture's full raw event history in chunks; accumulate until
`done: true`, then reconstruct the scrubbable timeline (section 5).

```json
{
  "type": "replay_chunk",
  "fixtureId": 14790158,
  "events": [
    { "fixtureId": 14790158, "action": "kickoff", "statusId": 2, "ts": 1721295000000, "seq": 1 },
    { "fixtureId": 14790158, "action": "high_danger_possession", "statusId": 4, "participant": 1, "ts": 1721295480000, "seq": 17 },
    { "fixtureId": 14790158, "action": "goal", "statusId": 4, "participant": 1, "ts": 1721296112000, "seq": 42 }
  ],
  "done": false
}
```

Each event has shape
`{ fixtureId, action, statusId, participant?, data?, ts, seq }` where `participant` is
`1 | 2` and `data` is an optional object (e.g. `{ "outcome": "OffTarget" }` on shots,
`{ "freeKickType": "Danger" }` on free kicks, `{ "outcome": "Overturned" }` on
`var_end`).

### Behavioral notes (all load-bearing)

- **Live match, no snapshot on subscribe:** after subscribing to an in-progress match
  you receive nothing until the next event fires (could be seconds). Render the scene
  in a graceful "connected, waiting for next update" state, using fixture metadata
  from the static fixtures file (section 6).
- **Finished match:** replay chunks arrive first, then `state` messages (which will
  just be the final unchanging state). Handle messages arriving in any order without
  crashing.
- **Scheduled match:** nothing arrives until kickoff. Show "waiting for kickoff" with
  the scheduled time.
- **Unknown message types:** ignore silently. (A `{ "type": "keyMoment", ... }`
  message type exists in the backend's type union; it may or may not be emitted. Do
  not depend on it ‚Äî the canonical key-moment detection is `keyMoments` growth in
  `state`.)
- **Unknown/extra fields:** tolerate them. Never throw on unexpected payload content.
- **Reconnection:** wrap the native WebSocket with auto-reconnect (exponential backoff,
  cap ~10s) and **re-send all active `subscribe` messages after every reconnect**.
  Show a small connection indicator in the HUD (section 10).
- **Uncovered fixtures:** not every fixture has data coverage. If a subscribed fixture
  produces no `state`/`replay_chunk` within ~10 seconds and its kickoff time is in the
  past, overlay a quiet notice ‚Äî "Live data coverage unavailable for this match" ‚Äî on
  top of the still-rendered empty pitch. Never render a broken/blank screen.

## 5. Replay Reconstruction (client-side reducer)

Live mode needs no reducer ‚Äî `state` messages are authoritative. But replay chunks
contain **raw events**, so scrubbing requires the frontend to reconstruct the state
timeline itself.

**Do this by porting the backend reducer verbatim.** It is a dependency-free pure
TypeScript module in this repo:

- `server/src/reducer/types.ts` ‚Äî `MatchState`, `RawScoreEvent`, `KeyMoment`, etc.
- `server/src/reducer/reducer.ts` ‚Äî `reduce(state, event) => newState`

Copy both files into the frontend unchanged (they import nothing but each other). This
guarantees replay reconstruction matches the backend's live output exactly. If the
backend tweaks constants, re-copy ‚Äî the server files are the source of truth.

For reference, the semantics you are porting (constants: `MOMENTUM_DECAY = 0.97`,
`MOMENTUM_STEP = 0.15`, `PRESSURE_STEP = 1`; danger weights: Safe 0.1, Attack 0.4,
Danger 0.8, HighDanger 1.2; danger‚Üízone: Safe‚Üídefensive, Attack‚Üímiddle,
Danger/HighDanger‚Üíattacking):

| Action | Momentum effect (toward `participant`) | Pressure effect | Other |
|---|---|---|---|
| `possession` | weight 0.1 | +0.1 to `defensive` | |
| `attack_possession` | weight 0.4 | +0.4 to `middle` | |
| `danger_possession` | weight 0.8 | +0.8 to `attacking` | |
| `high_danger_possession` | weight 1.2 | +1.2 to `attacking` | |
| `corner` | weight 0.4 | +1 to `attacking` | |
| `shot` | weight 1 (0.5 if `data.outcome === "OffTarget"`) | +weight to `attacking` | |
| `free_kick` | weight from `data.freeKickType` danger level (default Safe) | +weight to zone from danger level | |
| `goal` | weight 1.5 | none | score+1, appends `keyMoment` |
| `red_card` | weight 0.5 toward the **other** team | none | appends `keyMoment` |
| `var_end` | none | none | appends `var_overturned` keyMoment only if `data.outcome === "Overturned"` |
| `kickoff` / `halftime_finalised` / `game_finalised` | none | none | updates `statusId` |
| anything else | none | none | no-op |

Every event updates `lastTs`/`lastSeq`. Momentum update formula:
`clamp(momentum * 0.97 + direction * weight * 0.15, -1, 1)` where `direction` is `+1`
for participant1, `-1` for participant2.

**Reconstruction procedure:** starting from `initialMatchState(fixtureId)`, fold the
full event array in `seq` order, storing **every intermediate `MatchState`** in an
array of frames `{ ts, seq, state }`. Scrubbing = binary search this array by `ts`.
A full match is a few thousand events at most ‚Äî this is instant; do it once per replay
load, keep it in memory. Also derive the momentum-history polyline (section 8.5) from
these frames.

Recommended: a handful of unit tests on the ported reducer (a goal updates score, a
red card doesn't, momentum clamps) ‚Äî the only frontend tests worth writing this
weekend.

## 6. Data the Backend Does NOT Provide (you must supply)

The WebSocket carries no fixture list, no team names, no badges, no colors, and no
kickoff times. Two static data files close the gap. Never call TxLINE or any external
API from the browser to fill these.

### 6.1 `fixtures.json` ‚Äî fixture metadata

The match list and the mapping `participant1/participant2 ‚Üí real team` come from a
static file, hand-assembled at integration time (fixture IDs are exported from the
backend's database / TxLINE fixtures data by the backend side; build against the
format now with placeholder entries):

```json
[
  {
    "fixtureId": 14790158,
    "participant1": "France",
    "participant2": "Brazil",
    "kickoffUtc": "2026-07-19T19:00:00Z",
    "stage": "Final",
    "venue": "MetLife Stadium"
  }
]
```

`participant1`/`participant2` here bind, in order, to the contract's
`participant1`/`participant2` fields. `stage` is free text ("Group A", "Round of 32",
"Quarter-final", "Final", ...). Treat this file as replaceable content, not code ‚Äî
loading it should be one import.

### 6.2 `teams.ts` ‚Äî badge + color lookup

Build a static lookup for the **48 qualified teams** (2026 is the expanded 48-team /
104-match format ‚Äî if any older project doc says "32 teams", 48 is correct):

```ts
interface TeamMeta {
  name: string;        // matches fixtures.json team strings
  code: string;        // 3-letter, e.g. "FRA"
  primary: string;     // hex ‚Äî dominates that team's terrain/ribbon/glow
  secondary: string;   // hex ‚Äî accents, badge text
  aliases?: string[];  // e.g. ["USA", "United States", "United States of America"]
}
```

Lookup is by normalized name (case-insensitive, aliases checked). **Any unknown team
must resolve to a neutral fallback** (`code` = first 3 letters uppercased,
primary `#8A93A6`, secondary `#E6EAF2`) so nothing ever breaks on an unexpected name.

**Badges ‚Äî recommended approach:** these are national teams, so use **country flags in
a circular roundel** rather than sourcing crest artwork (federation crests are

Use the `flag-icons` npm package (MIT, self-contained SVGs ‚Äî satisfies "no external

secondary color, thin ring. Both read perfectly at scorebug size. Do not hotlink

## 7. Look and Feel ‚Äî Global

- **Aesthetic:** stadium at night. Near-black world; the pitch, terrain, and ribbon
  are the light sources. Everything important is emissive + bloom; nothing important
  is lit by scene lights.
- **Background:** `#05070C`, with gentle fog (same color, near ~80, far ~400) so the
  pitch apron fades into darkness. Optional cheap atmosphere: ~200 slow-drifting dust
  particles, and four stylized floodlight masts at the pitch corners (simple dark
  poles with a small emissive quad head ‚Äî no real shadow-casting lights).
- **Lighting:** low cool ambient (~0.25, slightly blue), one soft directional from
  above for gentle shading on the goal frames. The look comes from emissive materials
  plus a bloom post pass (threshold high enough that only the terrain peaks, ribbon,
  markers, and pitch lines bloom ‚Äî not the HUD, which is DOM, outside the canvas).
- **HUD chrome (2D):** dark glass panels `rgba(10,14,20,0.85)`, 1px `#2A3242`
  borders, white text `#E6EAF2`, muted text `#8A93A6`. Team colors appear only in
  team-bound elements (badge rings, scorebug name blocks, event-log chips).
- **Typography:** a condensed sans for broadcast feel (Barlow Condensed or Archivo ‚Äî
  self-hosted, not a CDN link) for the scorebug, takeover words, and labels;
  **tabular numerals** for score and clock so digits don't jitter. System sans for
  body text.
- **Motion rules:** nothing snaps. Terrain tweens (~1.2s ease-out on live updates,
  ~0.3s when scrubbing), ribbon head pulses, HUD values roll/flip. But motion is
  data-driven ‚Äî no idle animation that fakes activity when nothing is happening
  (exception: the breathing noise in 8.4, which scales with real pressure).

## 8. The 3D Scene (match view)

### 8.1 Coordinate system and units

1 scene unit = 1 meter. `y` up. Pitch plane at `y = 0`, centered on origin:

- `x ‚àà [-52.5, +52.5]` ‚Äî pitch length (105m).
- `z ‚àà [-34, +34]` ‚Äî pitch width (68m).
- **participant1 defends the `x = -52.5` goal and attacks toward `+x`.**
  participant2 mirrored. This orientation rule drives the terrain (8.4), the scorebug
  ordering (10), and the ribbon color coding (8.5). Never flip it.

### 8.2 Pitch geometry (FIFA-standard, this exact set of markings)

Dark surface: very dark desaturated green `#071B10`, with subtle alternating mowing
stripes along `x` (bands ~5.25m wide, alternate albedo `#0A2415`) ‚Äî low contrast, just
enough to read as grass at night. Beyond the touchlines, a ~6m darker apron, then fog.

White line markings, ~0.12m wide, slightly emissive (`#E8F1EA`, low intensity so they
glow faintly but don't bloom hard), rendered either as a generated texture on the
pitch plane or as flat geometry at `y = 0.01`:

| Marking | Spec |
|---|---|
| Boundary | full 105 √ó 68 rectangle |
| Halfway line | `x = 0`, full width |
| Center circle | radius 9.15 at origin, plus center spot |
| Penalty areas | 16.5m deep from each goal line, 40.32m wide, centered on `z = 0` |
| Goal areas | 5.5m deep, 18.32m wide, centered |
| Penalty spots | 11m from each goal line, on `z = 0` |
| Penalty arcs | radius 9.15 from each penalty spot, only the part outside the box |
| Corner arcs | radius 1m at all four corners |

**Goal frames** at both ends: 7.32m wide, 2.44m high, posts/crossbar ~0.12m thick,
white, slightly emissive. Simple box geometry, no net (or a very cheap translucent
net plane if trivially easy). The goals matter ‚Äî they make it read as a real pitch
from every camera angle.

### 8.3 Camera

- Perspective, FOV 45. Default position `(0, 55, 85)`, target `(0, 2, 0)` ‚Äî an
  elevated broadcast-gantry three-quarter view from the `+z` side. From here,
  participant1 attacks screen-right.
- OrbitControls: damping 0.08, `minDistance 25`, `maxDistance 160`, max polar angle
  ~80¬∞ (camera never goes below the pitch), pan disabled. Double-click resets to
  default with a ~0.8s tween.
- **Live idle drift:** after 20s without input in live mode, a very slow auto-orbit
  (~0.02 rad/s). Any input cancels. Gives the "ambient second screen" feel.
- **Replay fly-along (stretch goal, cut first if behind):** during replay auto-play, a
  "cinematic" toggle moves the camera along a path offset from the ribbon (10m to the
  `+z` side of the playhead, 4m above it, lookAt the playhead point), traveling with
  the playhead. Any user input exits back to orbit.

### 8.4 Pressure terrain

Two translucent heightfield surfaces ‚Äî one per team ‚Äî floating just above the pitch,
covering the pitch footprint (105 √ó 68), plane geometry subdivided ~128 √ó 64. Team1's
surface at base `y = 0.3`, team2's at `y = 0.6` (offset kills z-fighting where both
are near zero).

**Data ‚Üí height.** Each team contributes exactly 3 numbers per update
(`defensive/middle/attacking`). Map them to pitch thirds using the 8.1 orientation:

- participant1: defensive third `x ‚àà [-52.5, -17.5]`, middle `[-17.5, +17.5]`,
  attacking `[+17.5, +52.5]`.
- participant2: exact mirror (their attacking third is `x ‚àà [-52.5, -17.5]`).

So both teams' attacking mounds rise near the goal they are attacking ‚Äî pressure
visibly piles up on the goal under siege. This is the core image of the product.

Construction per vertex:

1. **Longitudinal profile:** treat the 3 zone values as control points at zone
   centers `x = -35, 0, +35` (team-oriented), interpolate smoothly between them
   (smoothstep or Catmull-Rom over the three points, clamped at the ends). No steps,
   no walls at third boundaries.
2. **Normalization:** `norm(v) = (v / maxSeen)^0.75`, where `maxSeen` is the maximum
   of all six zone values seen so far **in this match** (floor 1 to avoid divide by
   zero). The gamma lifts mid values. Using one shared max keeps the two teams'
   terrains honestly comparable. Note: values are accumulating totals, so late-game
   terrain shape changes slowly ‚Äî that's correct; recency lives in the ribbon. Do not
   add client-side decay.
3. **Height:** `y = base + norm * 6` (max terrain height 6m ‚Äî well under the ribbon
   band, section 8.5).
4. **Lateral envelope:** multiply by a cosine dome across `z` (1.0 at `z = 0`,
   ~0.4 at the touchlines) so each surface reads as a mound over the pitch, not a
   wall-to-wall slab.
5. **Breath:** add low-amplitude animated simplex noise ‚Äî amplitude 8% of the local
   height (so zero-pressure areas stay perfectly flat and never fake data), spatial
   scale ~10m, slow time scale. This makes high-pressure regions visibly "boil".

Since the whole field derives from 6 scalars + noise, do the displacement in the
vertex shader with the zone values as uniforms, and tween the uniforms CPU-side
(~1.2s ease-out per live update; ~0.3s when scrubbing). Cheap and perfectly smooth.

**Material:** team glow color (`primary`, or the `glow` override); transparent,
opacity ~0.35; emissive scaled by height (peaks glow bright and bloom, flats are
near-invisible); fresnel edge brightening; `depthWrite: false`, additive-leaning
blending so the two teams' surfaces mix where they overlap; render after the pitch.
An optional faint wireframe overlay (same color, opacity ~0.08) helps legibility of
the surface shape ‚Äî include if it looks good, drop if noisy.

**Ambient event feedback:** live `state` messages don't say "corner" or "shot" ‚Äî but
pressure deltas do. When a single update increases any one zone by a large step
(‚â• 0.9, i.e. a shot/corner-scale event), emit a radial ripple in the terrain: an
expanding ring (shader-driven, ~1.5s life, ~15m max radius) centered on that zone's
center, in that team's color. This makes the terrain feel connected to real match
events without inventing data.

### 8.5 Momentum ribbon (the timeline, and the replay scrub control)

The momentum-over-time visualization is a **3D ribbon inside the scene** ‚Äî not a 2D
chart, ever.

**Geometry and encoding:**

- The ribbon runs the length of the pitch, high above the terrain. **`x` = match
  time**: kickoff at `x = -52.5`, and the time domain `max(90 minutes, elapsed)` maps
  linearly across the full 105m (so the ribbon of an in-progress match grows
  rightward and a finished match spans the pitch; extra/stoppage time just rescales
  the mapping ‚Äî recompute the whole curve on each new sample, it's a few hundred
  points).
- **Vertical deflection = momentum**: centerline at `y = 12`, `y = 12 + momentum * 6`
  (so `y ‚àà [6, 18]`, always clear of the 6m terrain peaks). **Up = participant1
  dominant** (consistent with positive momentum = participant1).
- The ribbon is a **vertical band**: 1.4m tall, faces `¬±z` (double-sided), built from
  a Catmull-Rom curve through the momentum samples. From the default camera it reads
  as a glowing wave wall ‚Äî the match's heartbeat drawn in the air over the pitch.
- **Color is self-describing:** vertex color blends toward participant1's glow color
  as the ribbon rises above the centerline, participant2's as it dips below, passing
  through neutral white near zero. Emissive + bloom; opacity ~0.85.
- A thin dashed neutral guide line at `y = 12` spans the pitch length (the "level
  match" reference), with small tick marks + minute labels (`15'`, `HT`, `60'`,
  `90'`) as small 3D text sprites (drei `Text`). Numbers in 3D space are fine ‚Äî the
  2D-only rule is about panels and charts, not tick labels.

**Momentum samples:** in live mode, append `{ t: lastTs, m: momentum }` from every
`state` message; in replay, derive from the reconstructed frames (section 5).
Decimate to ‚â§ ~400 points (one per ~15s of match time) before curve fitting. If the
viewer joins a live match mid-way, history simply starts at the join point ‚Äî the
ribbon begins partway along; acceptable, do not fake backfill.

**Live head:** at the newest sample, a glowing marker ‚Äî a small bright sphere with a
pulsing halo (bloom does the work) and a short fading trail. The brightest object in
the scene. Behind it the ribbon persists at full brightness (history stays visible).

**Key-moment markers** sit ON the ribbon at each `keyMoment`'s timestamp:

- `goal`: a glowing torus/burst ring in the scoring team's color, ~1.2m.
- `red_card`: a card-proportioned red box (0.6 √ó 0.9m), slight tilt.
- `var_overturned`: an amber octahedron/diamond with a small "VAR" text sprite.

In replay these markers are clickable ‚Üí scrub jumps to that moment.

**Scrub interaction (replay only):** raycast the pointer against the ribbon (use an
invisible, wider proxy strip ‚Äî say 6m tall around the ribbon ‚Äî so it's easy to hit).

- Hover: a ghost playhead ‚Äî a thin vertical light beam from the ribbon down to the
  pitch, plus a floating minute label at the hover point.
- Click or drag: commits the playhead. The **entire scene** reflects the state at
  that time: terrain (0.3s tween), HUD score/clock, event log truncated to that
  point, ribbon segments **after** the playhead dimmed to ~25% opacity /
  desaturated, segments before at full brightness.
- Transport (HUD, section 10): play/pause auto-play at 10√ó/30√ó/60√ó real time
  (default 30√ó ‚Äî a full match in ~3 minutes), current-minute readout. Auto-play just
  advances the playhead; scrubbing and transport drive the same playhead value.

### 8.6 Live vs. replay ‚Äî visual differences

| | Live | Replay |
|---|---|---|
| HUD chip | pulsing red `LIVE` | neutral `REPLAY` (or `FT` chip + replay controls) |
| Ribbon | grows rightward; bright comet head; **not interactive** | full span; playhead beam; past bright / future dimmed; interactive scrub |
| Terrain | tweens on each incoming state (~1.2s) | tweens to playhead state (~0.3s) |
| Clock | ticks locally between updates (section 12) | shows playhead minute |
| Transport controls | hidden | visible |
| Camera | idle drift after 20s | no drift; optional fly-along during auto-play |
| Takeovers | fire on new keyMoments | fire during auto-play only (shortened, ~1.5s); **never** on manual scrub |

**Mode selection:** on entering the match view, subscribe; if `replay_chunk`s arrive
or the first `state` has `statusId === 100`, you're in replay mode. Otherwise live. If
a live match finalises while watching (`statusId` flips to 100), show the final
takeover if a goal coincided, then switch to replay mode in place (transport appears,
ribbon becomes interactive).

## 9. Key-Moment Takeover (full-screen)

The one sanctioned break from the ambient language ‚Äî a real broadcast beat, big and
unambiguous, for `goal`, `red_card`, `var_overturned`. This is a full-viewport DOM
overlay above the canvas (2D is allowed here), plus in-scene effects.

**Trigger:** a new entry in `keyMoments` (dedup by `type:seq` against a seen-set).
Queue takeovers if several arrive close together ‚Äî play sequentially, never overlap.
**Skippable:** any click or Escape dismisses instantly to the ambient scene.

**Timeline (~4.5s total):**

| Phase | Time | What happens |
|---|---|---|
| Impact | 0‚Äì400ms | In-scene: camera punch toward the relevant goal end (quick dolly/FOV kick, ~300ms) + a full-terrain shockwave ripple expanding from the attacking third. DOM: full-viewport flash in the team's primary color, opacity 0.9 ‚Üí 0. |
| Card | 400ms‚Äì3.5s | Full-screen overlay (below). |
| Release | 3.5‚Äì4.5s | Overlay exits with a fast vertical wipe; camera eases back to its prior framing; the permanent marker (8.5) appears on the ribbon with a small pop. |

**Card layouts** (all: centered column, condensed uppercase type, subtle animated
diagonal light streaks in the background, badge scales in with overshoot ease):

- **GOAL:** background = diagonal gradient of the scoring team's primary ‚Üí darkened
  primary. Huge team badge (roundel, ~28vh). Below it the word **GOAL** at massive
  scale (~18vw). Below: team name (~4vw), then the updated scoreline (~6vw, tabular)
  with the changed digit rolling/flipping in.
- **RED CARD:** background deep red (`#7A0A12` ‚Üí `#3A040A`) regardless of team. A
  card-proportioned red rectangle rotating in 3D CSS (~20vh tall). Words **RED CARD**
  (~10vw), then the offending team's name + badge, smaller.
- **VAR OVERTURNED:** background dark slate (`#0D1420` ‚Üí `#05070C`) with amber
  accents (`#FFB300`). **VAR** monogram large, then **DECISION OVERTURNED** (~6vw),
  then the affected team's badge + name.

During replay auto-play, use a compressed ~1.5s version (impact + brief card, no
camera punch). During manual scrubbing, no takeovers at all ‚Äî markers on the ribbon
carry that information.

## 11. Match List (`/`)

Functional entry point ‚Äî do not over-invest. Dark page consistent with the scene
palette. Three groups: **Live now**, **Upcoming**, **Finished** (order by kickoff).

## 12. Derived Values Cookbook

Things the contract doesn't hand you directly:

- **Match clock (live):** the state has no minute field. Anchor `t0` at the `ts` of
  the first `kickoff` event you observe (replay) or, if you joined mid-match and
  never saw kickoff, the fixture's `kickoffUtc`. Displayed minute
  `= (now ‚àí t0) / 60000`, minus 15 minutes once a `halftime_finalised` has been
  observed (statusId change works as a proxy if the action isn't visible live ‚Äî
  fall back to: subtract 15 when elapsed > 60 and momentum of events implies second
  half; if that's too fiddly, just show raw elapsed and cap display at `45+` / `90+`
  style). This is an approximation and that's acceptable ‚Äî label nothing to the
  second half-time precision; a broadcast-style `67:42` ticking locally between
  updates is the goal, not referee-grade accuracy.
- **Replay minute:** `(frame.ts ‚àí kickoffFrame.ts) / 60000`, same halftime
  adjustment using the `halftime_finalised` event in the log (replay *does* have it).
- **Momentum direction label:** never label it as a prediction. If you caption it at
  all (e.g. hover on the ribbon), the phrasing is descriptive: "France dominating" /
  "even contest".
- **`maxSeen` normalization** (8.4): reset per match, floor 1, monotonically
  non-decreasing; in replay, precompute per frame during reconstruction so scrubbing
  backward shows the max *as of that time* (this makes early-match terrain look
  correct instead of flattened by the full-match max).

## 13. Demo / Mock Mode (build this early ‚Äî it de-risks everything)

The backend may not be running (or no match may be live) while you build. Ship a
self-contained demo mode: `?demo=1` (or `VITE_DEMO=1`) bypasses the WebSocket and
loads `demo-match.json` ‚Äî a synthetic `RawScoreEvent[]` (~150‚Äì250 events) for a
fictional but plausible match: kickoff ‚Üí possession phases swinging between teams ‚Üí
corners/shots building one team's attacking pressure ‚Üí a goal around minute 25 ‚Üí
halftime ‚Üí a red card around 60 ‚Üí a comeback goal ‚Üí `game_finalised`.

Feed it through the exact same client pipeline as a real finished match
(reconstruction from section 5 ‚Üí replay mode), plus a "simulate live" toggle that
replays the events through the live-mode path at ~60√ó speed. Author the JSON by hand
following the event shape in section 4 and the action table in section 5.

This gives you: full development without the backend, a guaranteed-working demo for
the video, and a fallback if no covered fixture is live during recording. It is not
optional.

## 14. Recommended Tech Stack

These are recommendations ‚Äî swap any of them if you have stronger defaults, **except**
the 3D layer (R3F/three) and the hard constraints in section 2.

- **Framework:** Vite + React + TypeScript as a pure SPA (nothing here needs SSR;
  a viewer app with a separate WS backend is the classic SPA case). Next.js (app
  router, R3F components under `"use client"`) is fine too if you prefer it ‚Äî
  deployment target is Vercel/Netlify-style static hosting either way, with the WS
  URL injected via env var.
- **3D:** `three` + `@react-three/fiber` + `@react-three/drei` (OrbitControls, Text,
  Line helpers) + `@react-three/postprocessing` (Bloom ‚Äî central to the night-glow
  look; use a luminance threshold so only emissive elements bloom).
- **State:** `zustand`. One store: connection status, per-fixture latest `MatchState`,
  momentum history, replay frames, playhead, mode, takeover queue. R3F components
  read via selectors; per-frame interpolation happens in `useFrame` reading the store
  transiently (`store.getState()`) ‚Äî never trigger React re-renders per frame.
- **Styling:** Tailwind for the HUD and match list.
- **HUD/takeover animation:** framer-motion (or GSAP) for the takeover
  sequence and score-digit rolls; CSS transitions for everything smaller.
- **WS client:** native `WebSocket` in a ~50-line wrapper (backoff reconnect,
  resubscribe-on-reconnect, JSON parse with try/catch, typed message dispatch). No
  socket library needed.
- **Fonts:** self-hosted via `@fontsource/barlow-condensed` (or similar). No CDN
  requests.

## 15. Performance Budget

- Target 60fps on a mid-range laptop; clamp `devicePixelRatio` at 2.
- Terrain: vertex-shader displacement from ~6 scalar uniforms + noise ‚Äî never rebuild
  terrain geometry at runtime.
- Ribbon: rebuild only when a new momentum sample or playhead-dim change occurs
  (< 1/s in live), never per frame; ‚â§ ~400 curve samples.
- One scene, one canvas, one bloom pass. No shadows (shadow maps off). No
  environment maps. Pitch markings are one texture or static geometry.
- Dispose subscriptions/geometry on route change; unsubscribe fixtures you leave.

## 16. Non-Goals / Out of Scope (do not build)

- Anything betting/odds/prediction-adjacent (section 2 ‚Äî hard boundary, restated).
- Any wallet, token, transaction, or Solana UI ‚Äî including an "attested on-chain"
  badge; attestation is backend-only and has no frontend surface this weekend.
- Player-level data: player positions, dots on the pitch, individual heatmaps, lineup
  screens. The feed has no positional data; the zone-level terrain is the honest
  representation ‚Äî do not fake precision.
- User accounts, auth, preferences, favorites, notifications.
- Social/share features, chat, comments.
- Audio (commentary, crowd noise, sound effects).
- i18n ‚Äî English only.
- Deep mobile optimization. Desktop-first for the demo; it should load and not break
  on a phone (OrbitControls gives touch orbit for free; let the HUD stack/shrink
  responsively), but spend zero time on mobile-specific UX.
- Frontend test suites beyond a few unit tests on the ported reducer (section 5).
- More screens, settings panels, alternative visualizations, theme switchers.

fixture live or as replay. That demo-mode flow is exactly what gets screen-recorded
